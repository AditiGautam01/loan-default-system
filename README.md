## Business Problem

Banks lose billions yearly on loan defaults that could be flagged 60–90
days earlier. Existing rule-based systems miss subtle behavioural signals
buried across transaction history, bureau records, and payment patterns.

**This system:**
- Scores 307,000 applicants on default probability across 8 data sources
- Explains *why* each customer is high-risk using SHAP values
- Generates a plain-English risk brief via LLM — no data science jargon
- Runs automatically every Monday with drift detection and auto-retraining

## Quick Start

```bash
# 1. Clone and install
git clone https://github.com/YOUR_USERNAME/loan-default-system.git
cd loan-default-system
pip install -e ".[dev]"

# 2. Environment setup
cp .env.example .env
# Edit .env — add ANTHROPIC_API_KEY and DB_URL

# 3. Download dataset (Kaggle account required)
kaggle competitions download -c home-credit-default-risk
unzip home-credit-default-risk.zip -d data/raw/

# 4. Run full pipeline
python src/loan_default/data/loader.py     # Week 1 — EDA + feature store
python src/loan_default/models/trainer.py  # Week 2 — train + MLflow

# 5. Start API locally
uvicorn src.loan_default.api.app:app --reload
# Docs at http://localhost:8000/docs

# 6. Or run everything with Docker
docker-compose up --build
```

## Key Design Decisions

**XGBoost over neural networks** — Tabular financial data with ~180
features benefits more from gradient boosting. XGBoost + SHAP also
satisfies SR 11-7 model explainability requirements in banking.

**SHAP over feature importance** — SHAP gives per-prediction explanations.
A compliance officer can see *why* customer X was flagged, not just that
"loan_to_income matters globally."

**LLM for narratives** — SHAP values are numbers. Branch managers and
relationship officers need plain English. The LLM layer bridges that gap
without adding latency to the scoring pipeline.
